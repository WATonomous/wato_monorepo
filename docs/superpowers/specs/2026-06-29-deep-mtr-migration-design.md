# Deep MTR Repository Migration Design

## Goal

Move ownership of MTR-specific ROS inference scaffolding from `wato_monorepo` into
`WATonomous/deep_ros` without implementing working MTR inference. Preserve the existing
constant-velocity fallback as the default deployed behavior.

## Scope

This migration establishes repository ownership, ROS interfaces, build integration, and runtime
failure boundaries. It does not deliver a usable MTR model.

Included:

- Add model-neutral scene and trajectory-result interfaces to `deep_msgs`.
- Add a buildable `deep_mtr` package skeleton that follows `deep_core::DeepNodeBase` and backend
  plugin conventions.
- Remove the custom TensorRT backend abstraction from `prediction_ml`.
- Retain `prediction_ml` as the WATO-specific adapter, cache/selection policy, output converter,
  and constant-velocity fallback.
- Make the world-modeling image GPU-capable and include a pinned `deep_ros` revision.
- Add disabled-by-default launch wiring and migration-focused tests.
- Update `README.md` and `DEVELOPING.md` to reflect the new ownership boundary.

Explicitly excluded:

- Exporting MTR to ONNX.
- Defining or validating the final MTR tensor binding contract.
- Implementing agent-history tensor packing, map tensor packing, inference execution, or output
  decoding.
- Producing or distributing model artifacts.
- Enabling MTR in deployed launch configurations.
- Measuring MTR accuracy, latency, or GPU performance.
- Adding a native TensorRT `.engine` loader or TensorRT-specific backend.

Those excluded items remain the inference owner's follow-up work inside `deep_ros`.

## Decisions

### Repository boundary

`deep_ros` remains independent of WATO-specific `lanelet_msgs` and `world_model_msgs`.
Model-specific preprocessing, inference, and decoding belong to `deep_mtr`. WATO message
adaptation, fallback behavior, result freshness, and final world-model output remain in
`wato_monorepo`.

### Backend

The eventual MTR implementation will use the existing `deep_ort_gpu_backend_plugin` with
`Backend.execution_provider: "tensorrt"`. This migration does not add a native TensorRT backend.

### Runtime location

Both nodes run in the world-modeling Docker service. The image uses the CUDA base, clones a pinned
`deep_ros` revision, installs the same TensorRT runtime dependencies used by the perception image,
and receives GPU access through Docker Compose.

The existing world-modeling launch style uses standalone nodes. This migration preserves that
style. The typed ROS boundary permits later conversion to composition without changing ownership
or message semantics.

### Deployment default

MTR is disabled by default. The adapter always produces constant-velocity fallback output without
requiring `deep_mtr`, a model file, a backend, or a GPU. The `deep_mtr` node is launched only when an
explicit `enable_mtr` launch argument is true.

## Architecture

```text
tracked detections ─┐
ego pose ───────────┼─> prediction_ml adapter ─> constant-velocity fallback ─┐
lanelet context ────┘             │                                         │
                                  │ MtrScene                                │
                                  v                                         │
                           deep_mtr (deep_ros)                               │
                                  │                                         │
                                  v                                         │
                 ONNX Runtime GPU / TensorRT EP                             │
                                  │                                         │
                                  │ MtrPredictionArray                      │
                                  v                                         │
                         freshness + ID matching                            │
                                  │                                         │
                                  └─ fresh valid result replaces fallback ──┤
                                                                            v
                                                       world_model_msgs/WorldObjectArray
```

## ROS Interface Boundary

The migration adds semantic messages rather than model tensors. This leaves tensor layout and
binding choices to the inference owner while preventing `deep_ros` from depending on WATO message
packages.

### Scene request

`deep_msgs/MtrScene` contains:

- `std_msgs/Header header`: source timestamp and coordinate frame.
- `string request_id`: unique correlation ID generated by the adapter.
- `vision_msgs/Detection3DArray detections`: current tracked objects.
- `geometry_msgs/PoseStamped ego_pose` and `bool has_ego_pose`.
- `deep_msgs/MapPolyline[] map_polylines` and `bool has_map`.

`deep_msgs/MapPolyline` contains a semantic type and ordered `geometry_msgs/Point[]` coordinates.
The WATO adapter converts `lanelet_msgs/LaneletAhead` into this neutral representation.
`deep_mtr` owns temporal history and eventual conversion from repeated semantic scenes to MTR
tensors.

### Prediction result

`deep_msgs/MtrPredictionArray` contains:

- `std_msgs/Header header`: the source scene timestamp and frame, not completion time.
- `string request_id`: copied from the input scene.
- `deep_msgs/MtrObjectPrediction[] objects`.

Each object prediction contains its source track ID and zero or more modal trajectories. Each
trajectory contains a confidence and ordered poses. `prediction_ml` converts these neutral results
to `world_model_msgs::msg::Prediction`.

The adapter accepts results only when the request ID is known, the source timestamp is within the
configured TTL, the frame is expected, object IDs map to current tracks, trajectory lengths are
valid, and confidence values are finite and non-negative.

## Component Responsibilities

### `deep_ros/deep_msgs`

- Own the semantic MTR scene and result message definitions.
- Depend only on reusable ROS message packages.
- Generate interfaces required by both repositories.

### `deep_ros/deep_mtr`

- Provide the MTR lifecycle-node package skeleton.
- Use `DeepNodeBase` and the existing backend parameter schema.
- Subscribe to `MtrScene` and publish `MtrPredictionArray`.
- Expose clear source-file ownership seams for scene history/tensor preparation, inference, and
  result decoding.
- Compile and pass contract/lifecycle tests without claiming usable inference.

The skeleton must not fabricate successful model predictions. Until the inference implementation
exists, it publishes no MTR result and emits a bounded diagnostic when it receives a scene.

### `wato_monorepo/prediction_ml`

- Subscribe to tracked detections, ego pose, and lanelet context.
- Generate request IDs and publish neutral scenes only when MTR is enabled.
- Convert lanelets into neutral map polylines without defining MTR tensor layout.
- Cache accepted results by track ID and reject stale, malformed, or mismatched results.
- Generate fallback trajectories synchronously every tick.
- Replace a fallback trajectory only with a fresh accepted MTR result.
- Publish `world_model_msgs/WorldObjectArray` regardless of MTR availability.

The following current abstractions leave the monorepo because they duplicate `deep_ros` backend
responsibility:

- `IMtrInferenceEngine`
- `MtrMode::TensorRt` and compile-time backend selection
- `mtr_backend.hpp`
- `null_backend.cpp`
- `tensorrt_backend.cpp`
- `PREDICTION_ML_ENABLE_TENSORRT`

`MtrRuntime` is reduced or replaced with adapter-side request/result tracking. Model execution no
longer occurs in the monorepo process.

## Configuration

### Monorepo adapter parameters

- `mtr.enabled`: false by default.
- `mtr.request_topic`: neutral scene topic.
- `mtr.result_topic`: neutral result topic.
- `mtr.cache_ttl_s`: maximum accepted result age.
- Existing history/tensor/backend/model-path parameters are removed from the adapter.
- Existing fallback parameters remain unchanged.

### `deep_mtr` parameters

- Standard `DeepNodeBase` model and backend parameters.
- `Backend.plugin: "onnxruntime_gpu"` for eventual deployment.
- `Backend.execution_provider: "tensorrt"` for eventual deployment.
- Model-specific scene/tensor parameters are documented as owned by the inference follow-up and are
  not invented by this migration.

The monorepo documentation must refer to an ONNX model path and TensorRT execution provider. It
must not claim direct `.engine` loading.

## Docker and Cross-Repository Integration

The `deep_ros` change lands before the monorepo integration change.

1. Merge the `deep_msgs` interfaces and buildable `deep_mtr` skeleton in `deep_ros`.
2. Record the merged `deep_ros` commit SHA.
3. Change `docker/world_modeling.Dockerfile` to use the CUDA base and clone that exact SHA.
4. Install TensorRT runtime libraries in the world-modeling dependencies stage.
5. Give `world_modeling_bringup` NVIDIA runtime/device configuration in
   `modules/docker-compose.yaml`.
6. Copy or mount model assets using the existing `/opt/watonomous/models` convention, while
   leaving MTR launch disabled.

Pinning prevents an unrelated `deep_ros` main-branch change from silently changing the
world-modeling image. Development builds may override the ref explicitly, but the deployment
default remains immutable.

## Failure Handling

- With MTR disabled, `prediction_ml` never waits for or requires `deep_mtr`.
- A missing model, failed backend configuration, or failed `deep_mtr` lifecycle transition cannot
  stop fallback output.
- The adapter ignores unknown request IDs, stale timestamps, wrong frames, unknown track IDs,
  invalid confidence values, empty trajectories, and non-finite poses.
- Rejected results produce throttled diagnostics and do not replace fallback output.
- If `deep_mtr` stops publishing, cached results expire normally and fallback resumes per object.
- The placeholder `deep_mtr` node never publishes fabricated success data.

## Testing and Acceptance

### `deep_ros`

- Interface generation tests verify all new messages build.
- Contract tests verify the scene request correlation fields and neutral map representation.
- Lifecycle tests use the existing `deep_test`/mock-backend facilities; they do not require a GPU
  or real MTR model.
- A placeholder behavior test verifies that receiving a scene produces no prediction result.
- Existing `deep_ros` CPU/GPU tests continue to pass.

### `wato_monorepo`

- Unit tests verify lanelet-to-polyline adaptation without tensor packing.
- Unit tests verify request-ID correlation and per-result validation.
- Unit tests verify stale, malformed, wrong-frame, and unknown-track results are ignored.
- Existing fallback tests continue to prove deterministic output when MTR is disabled or silent.
- A launch test starts `prediction_ml` with `mtr.enabled: false` and no model/backend assets.
- The world-modeling Docker image builds with the pinned `deep_ros` revision.

### Acceptance criteria

- No custom TensorRT implementation remains in `prediction_ml`.
- `deep_ros` has no dependency on `lanelet_msgs` or `world_model_msgs`.
- The standalone `prediction_ml` launch publishes the same fallback output without a model or GPU
  inference. Integration into `world_modeling_bringup` remains disabled by default because the
  current live bringup does not launch either prediction implementation.
- Enabling the migration skeleton does not fabricate or publish MTR predictions.
- Documentation assigns future tensor packing, ONNX model integration, inference, and decoding to
  the `deep_mtr` inference owner.
- Working MTR accuracy, latency, and GPU execution are not acceptance criteria for this migration.

## Documentation Changes

`prediction_ml/README.md` remains user-facing and documents WATO topics, fallback behavior, the
disabled-by-default MTR bridge, and launch parameters. It removes direct TensorRT-engine examples.

`prediction_ml/DEVELOPING.md` documents the cross-repository architecture and changes its ownership
table:

- WATO adapter, fallback, result selection, and world-model conversion remain in the monorepo.
- Scene history/tensor preparation, backend integration, and decoding move to `deep_ros/deep_mtr`.
- The existing inference owner's task is explicitly preserved and linked to the `deep_ros` files.

`deep_mtr/README.md` states that the package is migration scaffolding and not a working model. Its
development documentation records the eventual ONNX Runtime/TensorRT-provider path without
promising model availability.
